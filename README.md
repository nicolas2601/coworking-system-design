# Diseño de Sistema · Plataforma de Coworking

Esta es mi solución a la **Prueba 3** (obligatoria) de la evaluación técnica: leer un
requerimiento de negocio ambiguo, estructurarlo en historias de usuario bien formadas y
traducirlo a un modelo de base de datos relacional coherente. Acá no se escribe código: se
escribe criterio.

El entregable completo está en **[`PRUEBA_3.md`](./PRUEBA_3.md)**, con las tres partes:

1. **Historias de usuario** — 14 historias (Operador 5 · Reservante 6 · Admin 3) con
   criterios de aceptación, estimación de complejidad y dependencias.
2. **Modelo de base de datos** — SQL `CREATE TABLE`, diagrama ERD (DSL de dbdiagram.io) y la
   justificación de cada decisión de diseño, incluyendo las preguntas P1–P4.
3. **Bonus** — mapeo del modelo a las capas de Clean Architecture.

## Cómo verlo

El documento es autocontenido en Markdown. Para el diagrama, pegá el bloque DSL de la
sección *Diagrama ERD* en [dbdiagram.io](https://dbdiagram.io) y exportá la imagen.
